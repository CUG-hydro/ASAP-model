# LLM Wiki

使用 LLM 构建个人知识库的一种模式。

这是一个思路文档，旨在被复制粘贴到你自己的 LLM Agent 中（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目标是传达高层级的思路，但具体细节由你的 Agent 与你协作完成。

## 核心思路

大多数人对 LLM 与文档的体验类似于 RAG：你上传一组文件，LLM 在查询时检索相关片段并生成答案。这虽然可行，但 LLM 每次都在从头重新发现知识。没有积累。当你提出一个需要综合五份文档的微妙问题时，LLM 必须每次都重新查找并拼凑相关片段。没有任何东西被沉淀下来。NotebookLM、ChatGPT 文件上传以及大多数 RAG 系统都是这样工作的。

这里的思路不同。LLM 不是仅在查询时从原始文档中检索，而是**渐进地构建并维护一个持久的 wiki** —— 一组结构化、互相链接的 markdown 文件，位于你和原始资料之间。当你添加新资料时，LLM 不会仅仅为后续检索而索引它。它会阅读它、提取关键信息，并将其整合到现有 wiki 中 —— 更新实体页面、修订主题摘要、标注新数据与旧主张相矛盾之处、强化或挑战不断演化的综合结论。知识被编译一次，然后**保持更新**，而不是每次查询时重新推导。

这是关键差异：**wiki 是一个持久的、可复利的产物。** 交叉引用已经存在，矛盾已经被标记，综合结论已经反映了你读过的所有内容。每添加一份资料、每提出一个问题，wiki 都会变得更加丰富。

你几乎从不需要亲自编写 wiki —— 全部由 LLM 编写和维护。你负责资料来源、探索方向和提出正确的问题。LLM 完成所有繁琐的工作 —— 总结、交叉引用、归档、整理 —— 正是这些让一个知识库随着时间真正变得有用。在实际使用中，我在一侧打开 LLM Agent，在另一侧打开 Obsidian。LLM 根据我们的对话进行编辑，我实时浏览结果 —— 跟踪链接、查看图谱视图、阅读更新的页面。Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。

这可以应用于许多不同的场景。例如：

- **个人**：追踪自己的目标、健康、心理、自我提升 —— 归档日记、文章、播客笔记，并随时间构建出关于自身的结构化画像。
- **研究**：在数周或数月内深入某个主题 —— 阅读论文、文章、报告，并渐进地构建一个包含不断演化论点的综合 wiki。
- **阅读一本书**：边读边归档每一章，为人物、主题、情节线索以及它们之间的联系建立页面。读完后你将拥有一个丰富的伴读 wiki。想想像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 wiki —— 数千个互相链接的页面，涵盖人物、地点、事件、语言，由一群志愿者花费多年构建。你可以在阅读过程中亲手构建这样的东西，让 LLM 完成所有交叉引用和维护工作。
- **商业/团队**：由 LLM 维护的内部 wiki，数据源包括 Slack 讨论、会议记录、项目文档、客户通话。可能由人类在回路中审核更新。Wiki 保持最新，是因为 LLM 完成了团队中没人愿意做的维护工作。
- **竞品分析、尽职调查、旅行规划、课程笔记、爱好深挖** —— 任何你随时间积累知识并希望它被组织起来而非零散分布的场景。

## 架构

分为三层：

**原始资料** —— 你策展的源文档集合。论文、文章、图片、数据文件。这些是不可变的 —— LLM 读取它们但从不修改它们。这是你的真理之源。

**Wiki** —— 一个由 LLM 生成的 markdown 文件目录。摘要、实体页面、概念页面、对比、概览、综合结论。LLM 完全拥有这一层。它创建页面、在新资料到达时更新它们、维护交叉引用，并保持一切一致。你阅读它；LLM 编写它。

**Schema** —— 一份文档（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），用于告诉 LLM wiki 的结构、约定以及在摄取资料、回答问题或维护 wiki 时应遵循的工作流。这是关键配置文件 —— 它使 LLM 成为一位纪律严明的 wiki 维护者，而非泛泛的聊天机器人。你和 LLM 会在使用过程中共同迭代它，逐步弄清什么对你的领域最有效。

## 操作

**摄取。** 你将一份新资料放入原始资料库，并告诉 LLM 去处理它。典型流程是：LLM 阅读资料，与你讨论关键要点，在 wiki 中编写摘要页面，更新索引，更新 wiki 中相关的实体和概念页面，并在日志中追加一条记录。单个资料可能会触及 10-15 个 wiki 页面。就个人而言，我更喜欢一次摄取一份资料并保持参与 —— 我阅读摘要、检查更新，并引导 LLM 重点关注什么。但你也可以在更少监督下一次性批量摄取多份资料。如何选择取决于你自己的风格，并应在 schema 中记录以便未来会话参考。

**查询。** 你针对 wiki 提问。LLM 搜索相关页面、阅读它们，并综合带引用的答案。根据问题的不同，答案可以采用不同形式 —— markdown 页面、对比表、幻灯片（Marp）、图表（matplotlib）、画布。重要的洞见是：**好的答案可以被归档回 wiki 中作为新页面。** 你请求过的对比、分析、你发现的关联 —— 这些都很有价值，不应消失在聊天记录中。这样你的探索就像摄取的资料一样在知识库中复利。

**Lint。** 定期让 LLM 对 wiki 进行健康检查。检查：页面间的矛盾、被新资料取代的过时主张、没有入链的孤立页面、被提及但缺少独立页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的数据空白。LLM 擅长建议新的研究问题和新的资料来源。这能让 wiki 在增长过程中保持健康。

## 索引与日志

有两个特殊文件帮助 LLM（和你）在 wiki 增长过程中进行导航。它们各有不同的用途：

**index.md** 是面向内容的。它是 wiki 中所有内容的目录 —— 每个页面附带链接、一句话摘要，以及可选的元数据（日期或来源数量）。按类别（实体、概念、资料等）组织。LLM 在每次摄取时更新它。在回答查询时，LLM 首先读取 index.md 以查找相关页面，然后深入阅读。在中等规模（~100 份资料、~数百个页面）下这种方法出奇地有效，并且无需基于嵌入的 RAG 基础设施。

**log.md** 是按时间顺序的。它是一个仅追加的记录，记录发生了什么、何时发生 —— 摄取、查询、lint。一个有用的提示：如果每条记录都以一致的前缀开头（例如 `## [2026-04-02] ingest | 文章标题`），那么该日志就可以用简单的 unix 工具解析 —— `grep "^## \[" log.md | tail -5` 可以获取最近 5 条记录。日志为你提供 wiki 演化的时间线，并帮助 LLM 理解最近完成了什么。

## 可选：CLI 工具

在某些时候你可能希望构建一些小工具，帮助 LLM 更高效地在 wiki 上操作。最明显的就是 wiki 页面的搜索引擎 —— 在小规模下索引文件已足够，但随着 wiki 增长，你需要一个真正的搜索引擎。[qmd](https://github.com/tobi/qmd) 是一个不错的选择：它是一个本地 markdown 文件搜索引擎，具有 BM25/向量混合搜索与 LLM 重排序，全部在本地运行。它既有 CLI（让 LLM 可以通过 shell 调用），也有 MCP 服务器（让 LLM 可以将其作为原生工具使用）。你也可以自己构建更简单的工具 —— LLM 可以帮助你在需要时快速编写一个朴素的搜索脚本。

## 技巧与诀窍

- **Obsidian Web Clipper** 是一个浏览器扩展，用于将网页文章转换为 markdown。对于快速将资料放入原始资料库非常有用。
- **本地下载图片。** 在 Obsidian 设置 → 文件与链接中，将"附件文件夹路径"设置为固定目录（例如 `raw/assets/`）。然后在设置 → 快捷键中，搜索"Download"以找到"为当前文件下载附件"，并绑定一个快捷键（例如 Ctrl+Shift+D）。剪辑完一篇文章后，按下该快捷键，所有图片都会被下载到本地磁盘。这是可选的，但很有用 —— 它让 LLM 可以直接查看和引用图片，而不必依赖可能失效的 URL。注意，LLM 不能在一遍之内原生读取带内联图片的 markdown —— 变通方法是让 LLM 先读取文本，然后再分别查看部分或全部被引用的图片以获得额外上下文。有点笨拙，但足够好用。
- **Obsidian 的图谱视图** 是观察 wiki 整体形态的最佳方式 —— 哪些页面相互连接、哪些是枢纽、哪些是孤儿。
- **Marp** 是一种基于 markdown 的幻灯片格式。Obsidian 有一个对应的插件。可以直接从 wiki 内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，用于在页面前置元数据上运行查询。如果你的 LLM 为 wiki 页面添加 YAML 前置元数据（标签、日期、来源数量等），Dataview 可以生成动态表格和列表。
- Wiki 本身就是一个 markdown 文件的 git 仓库。你可以免费获得版本历史、分支和协作能力。

## 为什么这有效

维护知识库中繁琐的部分不是阅读或思考 —— 而是整理。更新交叉引用、保持摘要最新、标注新数据与旧主张相矛盾之处、在数十个页面之间保持一致性。人类会放弃 wiki，是因为维护负担的增长速度超过价值。LLM 不会感到厌烦、不会忘记更新交叉引用，并且可以在一遍之内修改 15 个文件。Wiki 能够持续维护，是因为维护成本接近于零。

人类的职责是策展资料、引导分析、提出好问题，并思考这一切的意义。LLM 的职责是其他一切。

这个思路在精神上与 Vannevar Bush 在 1945 年提出的 Memex 相关 —— 一个个人的、策展的知识库，文档之间有关联式链接。Bush 的愿景比最终实现的网络更接近这个概念：私有的、主动策展的，文档之间的关联与文档本身同样重要。他没能解决的部分是"谁来维护"。LLM 解决了这个问题。

## 备注

本文档有意保持抽象。它描述的是思路而非具体实现。确切的目录结构、schema 约定、页面格式、工具 —— 所有这些都取决于你的领域、偏好和所使用的 LLM。上述所有内容都是可选且模块化的 —— 挑选有用的，忽略无用的。例如：你的资料可能只有文本，那就不需要任何图片处理；你的 wiki 可能小到索引文件就已足够，无需搜索引擎；你可能不关心幻灯片，只想要 markdown 页面；你可能想要一套完全不同的输出格式。正确使用这份文档的方式是：把它交给你的 LLM Agent，共同协作实例化一个适合你需求的版本。这份文档的唯一职责是传达这种模式。剩下的，LLM 可以自己搞定。
